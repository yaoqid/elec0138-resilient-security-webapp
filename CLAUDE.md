This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Flask-based hospital patient management system for UCL ELEC0138 Security & Privacy coursework. The project has two branches:
- **`master`**: Intentionally vulnerable version (CW1 — demonstrating attacks)
- **`secure`**: Hardened version with security fixes (CW2 — demonstrating defenses)

## Commands

### Setup
```bash
python -m venv .venv
source .venv/Scripts/activate   # Windows Git Bash
pip install -r requirements.txt
```

### Initialize & Seed Database
```bash
python init_db.py           # Creates instance/hospital_demo.db from instance/hospital_demo.sql
python generate_data.py     # Seeds 100 patients, 5 doctors, and medical records
```

### Run the App
```bash
cd app && flask run          # Starts on http://127.0.0.1:5000
```

### Attack Demonstration Scripts (CW1)
```bash
python scripts/test_sqli_login.py        # SQL injection POC against /login
python scripts/test_bruteforce_login.py  # Brute-force POC against /login
```

### View Audit Logs
```bash
python query_logs.py        # Prints login_logs table
```

## Architecture

**Single-file Flask app** (`app/app.py`) with no blueprints — all routes, auth, and DB access in one module.

- **Database**: Raw SQLite3 (no ORM). Schema defined in `instance/hospital_demo.sql`.
- **Auth**: Session-based (Flask `session` signed cookies). Decorators `@login_required` and `@role_required(*roles)` enforce access control.
- **Roles**: `admin`, `doctor`, `patient` — each gets a different dashboard route and template.
- **API**: Routes support both HTML form submissions and JSON (`Content-Type: application/json` or `?format=json`).
- **Audit logging**: `log_login_attempt()` records every login to `login_logs` table with automatic SQL injection pattern detection.

### Key Database Tables
`patients`, `doctors`, `medical_records`, `users` (with `role` and `linked_id` FK to patient/doctor), `login_logs`

### Intentional Vulnerabilities (master branch)
- **SQL injection** in `/login`: username/password concatenated directly into SQL query (CWE-89)
- **No rate limiting** or account lockout (CWE-307)
- **Plaintext passwords** (CWE-256)
- **Hardcoded `SECRET_KEY`**

### Security Fixes (secure branch)
Parameterized queries, Werkzeug password hashing, rate limiting, account lockout, secure session config.
